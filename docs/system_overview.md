# 技术监督问题分类系统介绍

## 1. 文档目的

本文档用于说明当前 `dw2.0` 集成项目已经实现的核心功能、功能实现原理、模型结构以及后续迭代方向，便于汇报、交接和后续维护。

当前项目围绕“技术监督问题识别与标签化”展开，已经完成第一版可运行基线（baseline），能够基于真实监督问题数据实现：

- 问题等级分类
- 问题类别分类
- 批量预测
- 人工复核
- 反馈回流
- 图谱对接导出
- 课题二知识图谱关联
- 标准修订需求识别
- 标准修订优先级排序

## 2. 当前版本定位

`dw2.0` 已经将原先拆分的两个模型侧版本集成到同一个项目目录内：

### 问题等级识别能力

继承自原 `dw1.1.2`，面向“问题等级”分类任务，输出标签：

- 一般
- 较大
- 重大

### 问题类别识别能力

继承自原 `dw1.2.5`，面向“问题类别”分类任务，当前输出标签：

- 设计与选型
- 采购与制造
- 安装与施工
- 调试与试验
- 验收与交接
- 资料与标识
- 运维与环境

### 集成后的决策链路

`dw2.0` 中 `configs/real_problem_level_v1.yaml` 已改为调用项目内置的类别模型：

- `configs/real_problem_category_v1.yaml`
- `artifacts/outputs_real_problem_category_v1/best_model`

因此运行决策增强时不再依赖外部 `dw1.2.5` 目录。

## 3. 已实现的核心功能

### 3.1 真实数据接入与清洗

系统已经支持从真实 Excel 数据中提取训练数据，并统一转换为模型训练使用的标准字段。

当前使用的原始数据文件位于：

```text
data/real/raw/pytest_processed_1.xlsx
```

数据清洗后，会生成：

- `pytest_processed_1_problem_level_v1.csv`
- `pytest_processed_1_problem_category_v1.csv`

### 3.2 训练与评估

系统支持使用配置文件驱动训练流程，自动完成：

- 数据切分
- 文本预处理
- 标签映射
- 模型训练
- 验证集评估
- 测试集评估
- 指标导出

当前导出的核心指标包括：

- 准确率（accuracy）
- 宏平均 F1（macro-f1）
- 加权平均 F1（weighted-f1）

同时还能导出：

- 分类报告（classification report）
- 混淆矩阵（confusion matrix）

### 3.3 单条与批量预测

系统支持：

- 单条文本预测
- 带结构化字段的预测
- 批量文件预测

批量预测结果中会自动附带：

- 预测标签
- 置信度
- 是否需要人工复核

### 3.4 人工复核表导出

系统能够把批量预测结果导出为适合人工检查的 Excel 复核表，包含：

- `summary`：汇总统计
- `need_review`：仅包含建议人工复核的样本
- `all_predictions`：包含全部预测样本

人工主要填写：

- `review_status`
- `reviewed_label`
- `review_comment`
- `feedback_time`

### 3.5 反馈回流

人工复核完成后，系统支持把复核结果重新整理成训练数据，再用于重新训练或增量训练。

回流后的数据会保留：

- 修正后的标签
- 原始样本标识
- 反馈来源标记

### 3.6 图谱对接导出

系统支持导出图谱侧可直接消费的 JSONL 文件（按行存放的 JSON 记录），一行表示一条结构化问题实例。

导出内容包括：

- 问题 ID
- 问题文本
- 问题等级 / 问题类别
- 设备类型
- 监督专业
- 监督环节
- 问题阶段
- 细则名称
- 最终确认标签

## 4. 功能实现原理

## 4.1 数据准备原理

原始 Excel 中包含大量中文业务字段，例如：

- 问题描述
- 问题等级
- 设备类型
- 问题产生阶段
- 问题产生环节
- 监督专业

系统首先将这些字段映射为统一英文列名，再做以下处理：

1. 统一清洗空值与异常值
2. 规范日期字段格式
3. 保留可用于训练的关键字段
4. 删除重复样本
5. 生成唯一 `sample_id`（样本编号）

这样可以保证训练、预测、复核、图谱导出都使用同一套字段标准。

## 4.2 模型输入构造原理

模型输入不一定只是“问题文本”本身。当前系统支持把结构化上下文和文本拼接成统一输入。

例如一条模型输入可以被组织为：

```text
设备类型: 主变压器；监督专业: 电气设备性能；监督环节: 竣工验收；问题文本: 主变压器预防性试验项目未按标准要求全部完成
```

这样做的目的，是让模型不仅看到问题描述，还能同时利用：

- 设备类型
- 专业领域
- 监督环节

从而提升分类稳定性。

## 4.3 训练原理

训练阶段整体流程如下：

```text
读取配置
-> 读取数据
-> 切分 train / valid / test
-> 文本预处理
-> 构建模型输入
-> tokenizer（分词器）编码
-> 创建分类模型
-> Trainer（训练器）训练
-> 输出指标与模型文件
```

为了保证实验可复现，系统会：

- 固定随机种子
- 固化 train/valid/test 切分
- 保存 `resolved_config.yaml`（实际生效配置）
- 保存 `label_mapping.json`（标签与数字编号映射）

## 4.4 人工复核原理

人工复核机制不是终端交互式逐条输入，而是采用“导出表格 -> 人工填写 -> 回流训练”的方式。

其流程为：

```text
模型批量预测
-> 生成 predictions_for_review.csv
-> 导出 predictions_for_review_human.xlsx
-> 人工填写 reviewed_label / review_status
-> 生成 retrain_dataset.csv
-> 再训练模型
```

其中：

- `attach_feedback_columns` 负责给预测结果补复核字段
- `build_retrain_dataset` 负责把人工确认后的结果整理成可回流训练数据

## 4.5 图谱导出原理

图谱导出不是直接导出模型内部向量，而是导出“结构化问题实例记录”。

每条记录包括：

- 问题文本
- 标签
- 设备
- 专业
- 阶段
- 细则
- 标签来源

图谱同学可以基于这些字段继续构建：

- 问题节点
- 问题等级节点
- 问题类别节点
- 设备类型节点
- 监督专业节点
- 标准细则节点

## 5. 模型结构说明

当前两个版本使用的基座模型均为：

```text
hfl/chinese-roberta-wwm-ext
```

它属于中文 BERT / RoBERTa 类预训练模型，结构为 **仅编码器 Transformer（Encoder-only Transformer）**，即只有编码器，没有解码器。

整体结构可以表示为：

```text
输入文本
-> Tokenizer
-> 词向量嵌入层（Embedding）
-> 12 层 Transformer 编码器（Transformer Encoder）
-> CLS 位置句向量表示
-> 随机失活层（Dropout）
-> 线性分类头（Linear）
-> 输出标签分数（logits）
```

### 主要结构参数

- 编码器层数：12 层
- 解码器层数：0 层
- 隐藏层维度（hidden size）：768
- 注意力头数（attention heads）：12
- 前馈层中间维度（intermediate size）：3072
- 当前最大输入长度配置：128

### 两个任务的输出头区别

#### 问题等级任务

问题等级三分类：

```text
Linear(768, 3)
```

#### 问题类别任务

问题类别七分类：

```text
Linear(768, 7)
```

也就是说，两个任务共享相同编码器底座，区别主要在最后一层分类头的输出维度不同。

## 6. 代码结构与职责划分

### 主流程

- `src/train.py`
  训练入口，负责组织完整训练流程

- `src/evaluate.py`
  评估入口

- `src/predict.py`
  预测入口

### 数据模块

- `src/data/prepare_real_data.py`
  真实数据清洗

- `src/data/prepare_problem_category_data.py`
  问题类别规则弱标注生成

- `src/data/dataset_builder.py`
  数据切分、标签映射、tokenize

- `src/data/preprocess.py`
  文本预处理与输入拼接

### 模型模块

- `src/models/factory.py`
  创建 tokenizer（分词器）和分类模型

- `src/models/interfaces.py`
  任务定义接口，预留多任务扩展

### 反馈与图谱模块

- `src/utils/feedback.py`
  复核字段补充与反馈回流

- `src/export_review_sheet.py`
  导出人工友好的 Excel 复核表

- `src/export_graph_records.py`
  导出图谱 JSONL 文件

## 7. 当前成果总结

截至目前，项目已经形成一个完整的第一版闭环：

```text
真实数据接入
-> 问题等级分类
-> 问题类别分类
-> 批量预测
-> 人工复核
-> 反馈回流
-> 图谱对接导出
```

当前工程已经不仅是一个单纯分类脚本，而是一个可训练、可评估、可预测、可复核、可回流、可对接图谱的基础框架。

## 8. 当前局限与后续方向

当前仍存在一些阶段性限制：

- 问题类别标签仍属于规则弱标注，不是最终人工高质量标注（gold label）
- 类别分布仍不均衡，少数类别样本偏少
- 当前仍是单任务单标签结构，多任务头尚未真正启用

后续可以继续推进的方向包括：

- 人工抽样修订问题类别标签体系
- 将问题等级与问题类别整合成多任务模型
- 引入更细粒度责任主体、人工复核建议等标签
- 增强与知识图谱的联动分析
